## 1) Security

### HIGH
- **S1 — Proyek soft-deleted tetap full-access (IDOR / broken lifecycle).** `middleware/rbac.ts:34-40` mengecek `project.deletedAt` tapi `deletedAt` **tidak di-`select`** → cast jadi `undefined`, cabang soft-delete tak pernah jalan. Setelah `softDeleteProject`, semua nested module (`/charter /cost /risk /schedule /attachments /audit /export /notifications`) masih bisa dibaca **dan ditulis**. **Fix:** tambah `deletedAt: true` ke select; treat soft-deleted sebagai `NotFound`.
- **S2 — Tidak ada rate-limiting auth.** `auth.routes.ts` `/login /register /refresh` tanpa `express-rate-limit`/lockout. bcrypt(12) memperlambat tapi tak menghentikan online guessing. **Fix:** limiter IP+akun + backoff/lockout.

### MEDIUM
- **S3 — Self-registration terbuka + directory penuh = harvesting email/role.** `POST /register` anonim bikin akun VIEWER hidup; `GET /users/directory` mengembalikan nama/email/role **semua** user aktif ke siapa pun yang login. Recon ideal untuk phishing + brute-force S2. **Fix:** matikan `/register` publik (pakai admin-create yang sudah ada), batasi field `/directory`.
- **S4 — Tidak ada revokasi token saat ganti password / lag role-change.** `auth.service.ts:67-79`: ganti password tak menginvalidasi refresh-token lama (TTL 7 hari) → reset pasca-kompromi tak benar-benar logout penyerang. Role downgrade juga baru efektif saat access-token (15m) kedaluwarsa. **Fix:** `tokenVersion`/`tokenInvalidBefore` per user, dicek di verify, di-bump saat ganti-password & deaktivasi.
- **S5 — Data master finansial terekspos ke semua role termasuk VIEWER.** `GET /ratecards` & `GET /resources` mengembalikan `unitCostPerManday` (+ nama/email user terkait) tanpa scoping role. **Fix:** gate field biaya ke FINANCE/PMO/ADMIN.

### LOW
- **S6 — `jwt.verify` tak mengunci algoritma** (`lib/jwt.ts:28-34`) — belum exploitable, tapi pin `{ algorithms: ['HS256'] }`.
- **S7 — Upload tanpa allowlist content-type** (`attachment.routes.ts:25`); MIME klien dipercaya. Mitigasi ada (cap 10MB, UUID key, `res.download` paksa attachment). **Fix:** `fileFilter` allowlist.
- **S8 — Kebijakan password tak konsisten di self-register** (`auth.schemas.ts:7` `min(8)` tanpa kompleksitas) vs `strongPassword` `min(10)` di tempat lain. **Fix:** pakai `strongPassword` di `registerSchema`.

### Dicek & AMAN
Raw SQL satu-satunya (`schedule.service.ts:207`) adalah tagged-template terparameterisasi (no injection); tak ada varian `*Unsafe`. RBAC: tiap nested route mutasi menumpuk `requireProjectAccess({write})` + `requireRole`. Self-lockout guard ada. Owner-binding attachment lintas-proyek ditegakkan. Secret prod unik & gitignored; error handler prod menyembunyikan internal.

## 2) Business-Process / PM-domain

### Bug nyata di logika
- **P1 [HIGH] — PV diukur dari plan saat ini, bukan baseline.** `calc/evm.ts plannedProgress()` + `schedule.service.ts:343-348` membangun `EvmTask` dari `planStart/planEnd`; `baselineStart/Finish` cuma dipakai untuk `finishVarianceDays`. Akibat: PM yang menggeser plan me-rebase PV → SV/SPI balik ke 0/1, slip jadi tak terlihat di indeks. **Fix:** umpankan baseline (fallback ke plan bila belum ada baseline) ke `plannedProgress`.
- **P2 [MED-HIGH] — Baseline cost & jadwal tak change-controlled.** Hanya narasi charter yang dikunci saat commit. Pasca CHARTERED, `cost.service.ts`/`schedule.service.ts` mengizinkan edit bebas direct/indirect/MR/tanggal tanpa CR & tanpa re-baseline (cuma `ensureChartered`). "Baseline" jadi sekadar indikatif. **Fix:** lewatkan perubahan baseline-affecting melalui workflow CR, atau minimal tandai baseline drift.
- **P3 [MED] — Risiko closed/occurred masih menambah reserve.** `calc/risk.ts contingencyReserve()` & `cost.service.ts loadRisksForReserve()` filter hanya `includeInReserve`, abaikan `Risk.status`. **Fix:** kecualikan status non-aktif; auto-clear `includeInReserve` saat CLOSED.
- **P4 [MED] — State machine status proyek tanpa guard transisi.** `updateProject` terima nilai apa pun dari `z.enum` (`projects.schemas.ts:25`) — bisa lompat CLOSED→DRAFT. **Fix:** validasi transisi yang diizinkan.
- **P5 [MED] — `budgetAtCompletion` ≠ BAC EVM (inkonsistensi nama/dok).** Kolom = baseline+MR (ERD menyebutnya "BAC"), tapi EVM pakai `costBaseline` (PMB, tanpa MR) sebagai `bac`. Split-nya disengaja, tapi penamaan/ERD bikin Finance salah rekonsiliasi. **Fix:** rename kolom → `totalBudget`, perbaiki baris ERD (tanpa ubah kalkulasi).
- **P6 [LOW-MED] — State CR `UNDER_REVIEW` tak terjangkau** — dibaca tapi tak pernah di-set. **Fix:** tambah transisi atau buang state.
- **P7 [LOW] — Version bump saat CR di-approve, sebelum re-commit** → label versi menyimpang dari snapshot terakhir; charter tinggal terbuka. **Fix:** bump versi saat commit.
- **P8 [LOW] — Margin portfolio pakai field manual (`costBaselineIdr`), bukan baseline terhitung** → bisa beda dengan modul Cost. **Fix:** turunkan dari baseline terhitung.

### Yang sudah benar (jangan "diperbaiki")
Set indeks/forecast EVM **lengkap & benar**: PV/EV/AC/CV/SV/CPI/SPI/BAC/EAC/ETC/VAC/TCPI + RAG NO_DATA-aware. EMV (threat +, opportunity −, clamp 0..1), residual EMV, P×I 1–25, heatmap 5×5 — benar. Contingency-in-baseline / MR-di-luar-PMB sesuai PMI. Model kapasitas (business-day spreading, cross-project keying) sehat. Deteksi siklus dependensi (3-color DFS) & roll-up 100%-rule benar.

## Kapabilitas yang kurang (gap fitur, bukan bug)

- **G1 [HIGH] — Tak ada critical path/float; dependensi inert.** `TaskDependency` simpan FS/SS/FF/SF+`lagDays`+deteksi siklus, tapi tak ada ES/EF/LS/LF, float, atau critical-path, dan dependensi tak membatasi tanggal — bahkan FS bisa dijadwalkan successor sebelum predecessor selesai **tanpa peringatan**. Minimal: validasi konsistensi tanggal dependensi + surfacing pelanggaran; lalu forward/backward pass.
- **G2 [MED] — Variance jadwal hanya finish & level-proyek** (`schedule.service.ts:330-335`). Tambah variance per-task & start-date.
- **G3 [MED] — Tak ada reserve drawdown/konsumsi.** Saat risiko terjadi, biayanya harus pindah reserve→AC; MR→baseline via CR disetujui. Tambah ledger reserve.
- **G4 [MED] — EVM "mengearn" contingency saat progres** (BAC termasuk contingency tapi EV/PV dibagi per direct-cost) → CPI/%complete tampak lebih baik dari kenyataan. Dokumentasikan caveat atau keluarkan contingency dari basis earnable.
- **G5 [MED] — CR tanpa dampak terkuantifikasi & tanpa traceability CR→versi.** Tambah field delta biaya/jadwal + link CR→`CharterVersion`.
- **G6 [MED] — Gap segregation-of-duties.** PM bisa set MR sendiri (`cost.routes.ts:18`), self-baseline charter (`charter.routes.ts:40`), actual-cost PM-writable (tak ada gate FINANCE). Ketatkan ke separasi PMO/PM/Finance.
- **G7 [LOW-MED] — Resource: deteksi saja**, tanpa leveling/auto-resolusi & tanpa kalender cuti/libur; manpower tak terjadwal dikecualikan dari over-allocation.
- **G8 [LOW] — Strategi respons risiko tak dibatasi jenis** (threat vs opportunity); residual parsial diam-diam diabaikan (butuh kedua input residual).
- **G9 [LOW] — Tak ada proses closure** (lock variance final, lessons-learned) & tak ada riwayat baseline berversi (cost/schedule baseline single in-place).
